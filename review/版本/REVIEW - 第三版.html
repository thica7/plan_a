5.2 各架构详细分析

L1 · 推荐保留

LangGraph 原生（plan_a 当前）

优点：DAG 显式、Send fan-out 并发、interrupt() HITL 一等公民、checkpointer 断点续跑、生态成熟（Langfuse/LangSmith 对接）

缺点：拓扑写死、子节点 ReAct 是手搓的（plan_a 1266 行的 collectors/logic.py 是怪兽）、状态都挤一个 GraphState

答辩话术：可放"DAG 可视化、可追溯"在 PPT C 位，A2 评分项护城河

L2 · 不推荐换

CrewAI

优点：Crew 抽象更高、Process(hierarchical) 内置 supervisor 模式、Task 描述自然语言友好

缺点：fan-out 并发支持弱、interrupt 不是一等公民、社区不如 LangGraph 活跃；换的话失去 plan_a 已投入工程纪律

L3 · CIMatrix 现状

自研同步 + EventBus

优点：零框架学习曲线、EventBus 解耦事件流（30+ 事件类型）、规则引擎 yaml 化、retry 直接 try/except

缺点：DAG 顺序硬编码 → 反馈环退化、无 fan-out 并发、无 checkpointer、可视化要自己画

答辩风险：评委问"为什么不用课题点名的框架"时要有备答（虽然课题已放开）

L4 · 部分借鉴

DeerFlow Plan-Execute

优点：supervisor 模式天然支持自适应任务拆分（B5 满分级别）、双层结构（顶层 plan-execute + 底层 research_team 子图）、Anthropic Claude Research 同款思想

缺点：messages 在 sub-agent 共享 history → token 消耗大、不适合 schema-first（输出非结构化）

plan_a 适配：把 planner 升级为 supervisor + Command(goto=...)，3-4 天能完成

L5 · CIMatrix 已部分实现

MiroFlow 双循环

优点：Plan-Execute 双层但 Plan 是规则驱动（不靠 LLM）、确定性高、Redis 状态持久化

缺点：缺真正 LLM 推理的 plan 能力、plan 规则要手写、不柔性

plan_a 借鉴：reflector 升级为类 MiroFlow 的"plan rules + LLM verifier"双轨

L6 · ★ 强推荐

Pydantic-AI Agent

优点：原生类型化 tool calling、自动 messages 隔离（修正 plan_a Gap-2）、Pydantic v2 校验贯穿、RunContext 注入依赖、社区热度上升中

缺点：缺 LangGraph 的 checkpointer/interrupt 等 graph-level 能力、还不够成熟

plan_a 适配：保留 LangGraph 顶层 + sub-agent 用 Pydantic-AI 替代手搓 ReAct —— 完美组合，3-4 天

L7 · 时间盒不允许

Temporal Workflow

优点：deterministic replay（C3 决策回放最强实现）、actor 风格、长时运行 + HITL 一等公民

缺点：要起 Temporal server、学习成本高、演示成本大；3 周时间盒不允许

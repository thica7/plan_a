# Backend Gap Analysis: RAG + Crawling + KB Optimization

Date: 2026-06-06
Scope: backend knowledge, crawler, LangGraph tools, API routes, shared schema, and docs under `F:/platform/competition/plan_a/`.
Role: Codex owns backend implementation. This report is analysis only; no implementation changes are proposed here as patches.

## 1. Current State Inventory

### Context and Documentation

- `.context/` was not present, so there were no local coding-style, workflow, or historical decision files to apply.
- `docs/rag-architecture.md` describes the intended subsystem: SQLite + FTS5 metadata, Qdrant vectors, bge-m3 embeddings, bge-reranker-v2-m3 reranking, httpx + trafilatura crawling, optional Playwright JS rendering, and LangGraph tools.
- `docs/api_contract.md` currently lists the older platform endpoints only. It does not include `/api/knowledge/*` or `/api/crawl/*`, so it is out of sync with the implemented routes.
- `backend/pyproject.toml` includes `qdrant-client` and `trafilatura`, but does not include embedding, reranker, Playwright, PDF, DOCX, CSV, or JSON parsing dependencies.
- There are no unit tests for `backend/packages/knowledge`, `backend/packages/crawler`, or the new LangGraph tools. Existing backend tests cover older run, health, observability, scoping, and event paths.
- The new RAG/crawler files are currently untracked in git, so the working tree appears to be mid-implementation.

## 4. Open Questions

1. Embedding runtime: should bge-m3 run locally in-process, through a local embedding service, or through a remote model endpoint? What batching limits, GPU assumptions, and timeout budget should Codex target?
2. Reranker runtime: should bge-reranker-v2-m3 be mandatory for production search, optional behind a feature flag, or invoked only above a query/result threshold?
3. Chunking policy: should chunks target tokens rather than characters, and what defaults are expected for chunk size, overlap, heading preservation, table handling, and multilingual segmentation?
4. Deduplication policy: is exact content hash enough, or do we need near-duplicate detection by canonical URL, SimHash/MinHash, embedding similarity, or source-specific thresholds?
5. Retention/freshness: how long should crawled content remain active, and what recrawl cadence should apply by source type, competitor, or dimension?
6. Source trust model: which `source_type` values are canonical, and should retrieval boost official pages, reports, or manually uploaded sources?
7. File storage: should original uploaded/downloaded files be persisted under `runs/`, object storage, or not retained after parsing?
8. Crawl scope: should the crawler stay limited to explicitly supplied URLs, or should it expand links within same-domain/sitemap/depth budgets?
9. API surface: should `/knowledge/crawl-jobs` and `/crawl/jobs` both remain, or should one route own crawl job lifecycle to avoid duplicated logic?
10. Deployment target: is SQLite intended to remain for the 50k document baseline, or should Phase 5 plan a PostgreSQL migration as described in `docs/rag-architecture.md`?

## 5. Risks

| Risk | Impact | Mitigation |
|---|---|---|
| Retrieval silently runs sparse-only in production | Analysts may believe they are using RAG with bge-m3/Qdrant while dense search and reranking are inactive. | Add explicit provider wiring and health flags. Make `/api/knowledge/search` report retrieval mode and fail or warn when dense/rerank providers are unavailable. |
| SQLite contention and shared connection lifecycle | API routes and tools can close a class-level repository connection unexpectedly; concurrent writes may hit locks at scale. | Move to app-managed repository lifecycle, enable WAL/busy timeout, keep write transactions short, add batch transactions, and test concurrent ingest/search. |
| Crawler can fetch unsafe or out-of-scope targets | Current denial only covers literal localhost names and does not resolve DNS/IP ranges. | Add DNS resolution checks for private/link-local/metadata IPs, configurable allowlists, max redirects with per-hop validation, and tests for SSRF cases. |
| Crawl jobs do not populate the KB | UI/API may show successful crawls but no searchable knowledge, breaking the intended crawl -> parse -> dedupe -> ingest pipeline. | Make crawl result ingestion part of `_run_crawl_job`, record `document_id`, and add an acceptance test that a successful crawl becomes searchable. |
| Parser/file format expansion can create inconsistent chunks and citations | Different parsers may produce low-quality text, lost tables, or citations without stable source spans. | Define a common `ParsedDocument` contract with parser version, offsets, page/table metadata, normalized text, and warnings before adding new file handlers. |

## Recommendation

The backend has a useful Phase 1 skeleton: models, SQLite/FTS metadata, Qdrant adapter, ingestion pipeline, retrieval orchestrator, crawler primitives, tools, and FastAPI routes all exist. The critical gap is not file volume but missing production wiring: embeddings, reranking, crawl-to-ingest, durable crawler state, file parsers, and SQLite/Qdrant scale hardening. Phase 2 should first make the existing path truthful and durable: stabilize repository lifecycle, add chunk-level sparse search, wire bge-m3/reranker providers, ingest successful crawls, and add focused tests. Phase 3 can then expand formats and multi-source crawling without building on a sparse-only or non-durable foundation.
